## Source of Truth

| Data Domain | Source of Truth | Cache Layer |
|---|---|---|
| User identity and authentication | PostgreSQL (users, refresh_tokens) | Redis (session metadata) |
| Learning progress | PostgreSQL (lesson_attempts, enrollments) | Redis (dashboard cache, 60s TTL) |

| Notifications | PostgreSQL (notifications) | Redis (unread count) |
| Audit logs | PostgreSQL (audit_logs) | — |

## Timestamp Rules

- All timestamps stored in UTC (TIMESTAMPTZ in PostgreSQL)
- `created_at`: set on INSERT, never updated
- `updated_at`: set on INSERT and every UPDATE (trigger or ORM hook)
- `deleted_at`: set on soft delete; NULL means active
- Client receives timestamps in ISO 8601 format: `2024-01-01T00:00:00Z`
- Streak calculation uses school timezone (stored in schools.timezone)

---

## IDs and Public References

- All primary keys: UUID v4 (not sequential integers — prevents enumeration attacks)
- UUIDs generated by the application layer (not the database)
- No sequential IDs exposed in public API responses
- Match IDs used in WebSocket URLs are UUIDs (not guessable)

---

## Data Integrity Rules

- Foreign key constraints enforced at database level
- Enum values enforced at database level (PostgreSQL native enums)
- Check constraints for grade (1–12), semester (1–2), difficulty (1–5)
- Unique constraints: user email, school code, question version per question
- Partial unique indexes for active-only uniqueness (e.g., active class enrollment)
- All schema changes via migrations only (no direct DDL in production)

## Data Lifecycle States

### Question Lifecycle
```
draft → pending_review → published → archived
         ↑                    ↓
         └──── (edit) ────────┘ (creates new version)
```

### User Account Lifecycle
```
active → suspended → deactivated → (soft deleted) → (hard deleted after 30 days)
pending_parental_consent → active (on consent)
```

### Match Lifecycle
```
pending → active → completed
       → cancelled (no acceptance within 60s)
       → forfeited (anti-cheat or disconnect)
```

### Assignment Lifecycle
```
draft → published → closed
```

## Retention and Archiving

| Data Type | Active Retention | Archive | Hard Delete |
|---|---|---|---|
| Student learning records | Duration of enrollment | +1 year in cold storage | After archive period |
| Match history | 2 years in primary DB | — | After 2 years |
| Audit logs | 1 year in primary DB | 6 years in cold storage | After 7 years total |
| AI generation logs | 90 days | — | After 90 days |
| Notifications | 90 days | — | After 90 days |
| Refresh tokens | 30 days (expiry) | — | On expiry |
| Soft-deleted users | 30 days | — | Hard delete after 30 days |

## Storage Rules

| Data Type | Storage Location | Encryption |
|---|---|---|
| Structured data | PostgreSQL | AES-256 at rest (cloud provider managed) |
| Session/cache data | Redis | In-memory; Redis AUTH required; no sensitive PII |
| Uploaded documents (import) | Cloud Storage (S3/GCS) | AES-256 at rest; private bucket; signed URLs |
| Exported reports | Cloud Storage | AES-256 at rest; signed URLs with 1-hour expiry |
| Static assets (frontend) | CDN | Public |
| AI prompt/response logs | PostgreSQL (ai_logs table) | AES-256 at rest |

## Analytics Architecture

### Phase 1
- Analytics events written to `analytics_events` table in PostgreSQL
- Event schema: `{ event_type, user_id, school_id, properties (JSONB), occurred_at }`
- Dashboard queries run against PostgreSQL with read replica
- No separate analytics store in Phase 1

### Phase 2
- Analytics events streamed to a dedicated analytics store (ClickHouse or BigQuery)
- Real-time aggregations for teacher and admin dashboards
- Retention: 2 years in analytics store

### Key Analytics Events

| Event | Properties |
|---|---|
| lesson_started | lesson_id, student_id, grade, topic |
| lesson_completed | lesson_id, student_id, score, xp_earned, duration_seconds |
| match_started | match_id, player1_id, player2_id, grade |
| match_completed | match_id, winner_id, player1_score, player2_score |
| question_answered | question_id, student_id, correct, time_seconds, hint_used |
| achievement_earned | achievement_id, student_id |
| ai_recommendation_clicked | recommendation_id, student_id, action |
| teacher_dashboard_viewed | teacher_id, class_id |

## Data Quality and Governance

- All required fields enforced at database level (NOT NULL constraints)
- Enum values enforced at database level
- Application-level validation runs before database writes
- Data quality checks run as part of import pipeline (missing fields, invalid values)
- Duplicate detection runs on question import and AI generation
- Audit logs provide full change history for all mutable entities
